Git log

May 22, 2020 3-minute read

On normal winter days of January, planning for Google summer of code 2018, targeting organizations, narrowing my projects that I want to contribute to.

Luckily, Delhi University’s credible college Cluster Innovation Center initiated a programme “Cluster Month of Code” abbreviated as CMoC for the whole month of February 2018.

Mostly Inspired by IIIT allahabad’s programme Opencode, CIC took this initiative and I registered, as I did to OpenCode. But due to lack of my time management skills, which is improving daily, I couldn’t dedicate my time to any project and thus, missed opencode opportunity.

Fortunately, I managed to work along CMoC on the assigned project of GIT, a very credible distributed version control system written by the father of linux kernel himself in very few days. [Link to its history].

Fortunately, git has good amount of documentation to read, to learn more about. They discuss everything on their mailing lists and are very active on their irc channel #git on freenode server as well.

My Journey:

\1. Subscribe to Mailing list as the first step to discuss, be part of the contribution discussion. That’s the very first step to be a part of an open source project contribution, watching the discussion, and be a part of the discussion.

For the mailing list, it might not be easy for someone new to this field, because every mailing list especially, upstream projects have certain guidelines that need to be followed. These guidelines are very important for the conversation to be conducted in controlled manner so that it would be easy for others to read and understand the mail.

Though, each mailing list has different guidelines, but there are certain guidelines that are universal, that everyone should know about and for this, read the mailing list guidelines by shaktimaan.[link to his guidelines.]

Short summary of the mailing list guidelines:

* Don’t use messaging language, use full word and write the message in words as simple as possible.

* Never do a top posting on a mailing list, because it creates a discontinuity in the flow of text.

* Be patient for someone to answer and be humble as possible. After all, no one is working for anyone and still helping you.

\2. Read the project developer guide, documentation and contribution guidelines.

\3. Fork/Clone the project repository on your system and set up the working environment for you to work on.

\4. Install the software from the sources keeping the system version intact to not affect your daily work.

NOTE: Careful while installing any software from sources, and try not to break the system installed software to keep your work unaffected.

I installed git in /usr/local/bin directory so that, my global system installed git will not be affected.

But first of all, to install any software from source in linux, you need to install dev tools using: `sudo apt install libssl-dev &&

sudo apt-get install libcurl4-openssl-dev

` (Don’t copy paste any linux commands from third-party sites.[link to my blog]

Only then, you will be able to install any software from source.

PS: This was my first challenge towards contributing to any upstream project.

\4. Try to understand the project structure and code. Ask for help if needed. This step takes some time, because having an ability to read boring yet very important documentation is a bliss and takes time to acquire.

\5. NEVER QUIT! I understand, it happened to me too. You might feel frustated, and lose your motivation to work on it any longer. But don’t quit so easily, if it would be that easy, everyone would be doing it :P.

Quoting my friend, learning is never easy but can be fun if you feel motivated enough towards it.
